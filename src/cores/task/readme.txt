# task

该目录实现 **任务级处理单元（Task-level Processing Element）** 及相关辅助结构，用于描述任务驱动型计算节点、任务信息以及功耗跟踪逻辑。

## 目录职责

`task` 目录中的模块主要负责：

- 建模任务驱动的处理单元（PE）
- 管理任务描述信息
- 在任务执行过程中产生/接收通信流量
- 记录或估算任务执行相关的功耗信息

## 主要文件

### `TaskPE.ned`
任务处理单元 `TaskPE` 的 NED 结构定义文件。  
用于描述该模块在 OMNeT++ 中的参数、接口和组成结构。

### `TaskPE.h`
`TaskPE` 类声明文件。  
定义任务处理单元的状态、任务队列、执行逻辑接口等。

### `TaskPE.cc`
`TaskPE` 实现文件。  
负责任务调度、任务执行、与网络通信交互等核心行为。

### `TaskDescriptor.h`
任务描述结构定义文件。  
用于表示单个任务的属性，例如任务 ID、通信需求、执行特征等。

### `PowerTrace.h`
功耗跟踪相关类或数据结构声明文件。

### `PowerTrace.cc`
功耗跟踪功能实现文件。  
用于记录任务运行过程中的功耗信息，支持能耗分析。

## 模块定位

该目录比普通 `sources` / `sinks` 更偏向**应用层/任务层建模**，其作用通常包括：

- 把“节点”从简单的流量注入器扩展为“可执行任务的处理单元”
- 将计算行为与通信行为结合起来
- 支持做任务映射、调度、能耗与通信协同分析

典型关系如下：

`TaskPE -> Source/Sink/NI -> NoC`

或者更准确地说：

- `TaskPE` 产生任务和通信需求
- `NI` 负责将这些通信需求接入网络
- 网络返回结果后再由任务模块继续处理

## 设计特点

- 面向任务级仿真，而不仅仅是纯随机流量
- 更适合研究：
  - 任务映射
  - 任务调度
  - 通信-计算协同优化
  - 功耗/能耗分析
- 可作为更高层应用模型与 NoC 之间的桥梁

## 备注

如果后续你要把 HNOCS 扩展成支持 ONoC 或可重构拓扑的平台，那么 `task` 目录会非常重要，因为它更接近真实应用工作负载，而不仅仅是人工流量源。
